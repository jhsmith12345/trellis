First: Auth + Database + Voice Agent (the core loop)
Set up Firebase Auth and Cloud SQL with just the tables you need: clients, conversations, transcripts. Build the voice agent on Cloud Run with Gemini Live API. Get one person to create an account, talk to the agent, have their conversation recorded and transcribed, see the data land in the database. This proves the hardest piece works. The "portal" at this stage is just a login screen and a "talk to agent" button. Ugly is fine.
Second: Insurance card upload + VOB
Now that you have a client record being populated by the voice agent, add the ability to upload insurance card images. Gemini Vision extracts the data. Wire up VOB — manual task queue first, Stedi later. At this point you have a qualified lead: name, presenting concerns, insurance verified.
Third: BD dashboard + lead pipeline + financial agreement form
This is the first real "portal" that needs to look good because your BD person will live in it. It reads from the client data and VOB results that already exist. The financial agreement request form is a new write that triggers the next step. This is where the human-in-the-loop enters the workflow.
Fourth: E-sign flow
BD submits the form, documents generate and get sent to the client. Client signs. This needs the Google Docs template system, the Cloud Run signing page, the blockchain write. When the client finishes signing, their status flips to "ready for intake."
Fifth: Scheduling + clinician intake package
Calendar integration. When documents are signed, intake gets scheduled. The clinician gets the full package: voice agent transcripts, BD notes, insurance info, signed documents. This is the clinician's first touchpoint with the system, so this portal view matters.
Sixth: Session recording + transcription + AI notes
Meet recording detection, Speech-to-Text (reusing the pipeline you already built for the voice agent), Gemini note generation. Clinician reviews in Docs.
Seventh: Note signing + billing cascade
Clinician signs the note, claim assembles, CMS-1500 generates. Manual billing first. Automated clearinghouse submission later.
Eighth: Client portal, clinician dashboard, ledger, payments

Design Principles
Every component is built in the order a real person encounters it. A potential client discovers the practice, creates an account, talks to the AI agent (which is recorded from the first word), decides to pursue care, onboards, has their first session, gets a note, gets billed, and gets paid. The clinician configures their practice, sees a new client appear, conducts a session, reviews a note, signs it, and watches the money arrive.
This document maps every component, the event that triggers it, what it produces, and what it hands off to next.

Key Architecture Decisions
Auth-first voice agent. No anonymous conversations. Every interaction with the voice agent requires a logged-in account. This means every word spoken is tied to a client_id from the start, every conversation is recorded and transcribed, and the recording pipeline is shared infrastructure used by both intake conversations and therapy sessions.
Immediate record collection. The moment someone creates an account and speaks to the agent, we are capturing data. The voice agent conversation transcript becomes the first clinical artifact in their record. No wasted interactions. No data that exists outside the system.
BD human-in-the-loop for IOP/PHP. The voice agent qualifies leads and collects data. But the decision to admit to IOP/PHP requires human judgment: level-of-care assessment, rapport building, nuanced financial discussion, program fit evaluation. The system delivers a warm lead package to BD with everything the agent collected plus VOB results. BD does the close. Submitting the Financial Agreement Request Form is the trigger that sends paperwork to the client and initiates the formal admission process.
Landing page is static, agent is behind auth. The public-facing practice page shows practice info, specialties, insurance accepted, and a clear call to action: create an account to get started. General practice questions are answered by the static page content. The voice agent is the reward for signing up.
Shared recording pipeline. The same Speech-to-Text, transcription, and storage infrastructure handles voice agent intake conversations and therapy session recordings. Build it once in Phase 2, use it everywhere.
PHASE 1
Foundation & Practice Setup
Before anyone can use the system, the clinician needs infrastructure and a configured practice.
Component 1.1: GCP Project Provisioning (Terraform)
Automated deployment of the entire infrastructure stack into a new GCP project. This is the very first thing that happens. One command spins up everything: Cloud SQL (PostgreSQL), Cloud Run services, Firebase Auth, Drive folder structure, service accounts, IAM roles, network configs, and secrets. The clinician (or we, on their behalf) runs terraform apply and has a working backend in minutes.
Triggered by: New practice signup / self-deploy from open source repo
Outputs: Running GCP project with all services, database schema deployed, Cloud Run endpoints live
Tech: Terraform templates, Cloud SQL db-f1-micro, Cloud Run, Firebase Auth, Secret Manager
Component 1.2: Clinician Onboarding & Practice Profile
The clinician signs in for the first time and configures their practice. This includes: practice name, address, phone, website, NPI number, license type and number, taxonomy code, EIN/SSN, malpractice insurance info, specialties, treatment modalities, populations served, accepted insurance payers, fee schedule (per CPT code per payer), telehealth availability, office hours, cancellation policy, practice philosophy/bio. This data feeds the landing page (so visitors see practice info), the voice agent (so it has conversational knowledge), and the billing engine (so it knows rates and payer info). Also sets up Google Workspace integration: connects Calendar, configures Drive folder structure, sets up Meet recording detection.
Triggered by: First clinician login after provisioning
Outputs: Practice profile in Cloud SQL, landing page content populated, voice agent knowledge base ready, fee schedule configured, Google Workspace connected
Tech: Cloud Run admin UI, Cloud SQL, Google Calendar API, Drive API
Component 1.3: RBAC & User Management
Role-based access control for all system users. Roles: Practice Owner (full access), Clinician (own clients + notes), Supervisor (own clients + supervisee notes, co-sign capability), Business Development (lead pipeline, intake data, VOB results, financial agreement forms, NO clinical notes after admission), Billing Staff (financial data, no clinical notes), Client (own portal data only). Each role defines exactly what data they can see, create, edit. Supervisors can view supervisee clinical notes but not other client records. BD can see intake data and voice agent transcripts but not therapy session notes. All access is logged to the audit trail. This component supports the supervised associate model where a supervisor oversees 5-10 pre-licensed clinicians.
Triggered by: Practice profile setup complete
Outputs: Auth tokens with role claims, access control middleware on all endpoints
Tech: Firebase Auth custom claims, Cloud Run middleware, Cloud SQL role tables
Component 1.4: Audit Trail & PHI Access Logging
Append-only audit log capturing every access to protected health information. Who accessed what, when, from where (IP, user agent), what action was taken. This runs from day one, before any client data enters the system. Every API call that touches PHI writes to this log. The log itself is immutable (INSERT only, no UPDATE or DELETE, enforced by PostgreSQL trigger). Required for HIPAA compliance and becomes the backbone of security auditing as a premium service.
Triggered by: System initialization (always active)
Outputs: Append-only audit_log table in Cloud SQL
Tech: PostgreSQL trigger blocking UPDATE/DELETE on audit_log, structured logging to Cloud Logging
PHASE 2
First Contact & Recording Infrastructure
The public presence goes live. The recording and transcription pipeline is built here because the voice agent needs it from day one.
Component 2.1: Practice Landing Page (Public, No Auth)
A lightweight static web app hosted on Cloud Run. The public-facing landing page for the practice. Shows practice info pulled from the practice profile: clinician bio, specialties, accepted insurance, office hours, telehealth availability, contact info, cancellation policy. Prominent calls to action: "Create an account to get started" and "Already have an account? Sign in." This is the only public-facing page. Everything else requires authentication. General FAQ content (what to expect in therapy, how insurance works, what telehealth looks like) can be included as static content so common questions are answered without needing the voice agent.
Triggered by: Practice profile configured (Component 1.2)
Outputs: Public URL (practice-name.run.app), practice info displayed, signup/login entry points
Tech: Cloud Run static SPA, reads practice profile from API, responsive mobile-first design
Component 2.2: Recording & Transcription Pipeline (Shared Infrastructure)
The core audio processing pipeline used by BOTH the voice agent and therapy sessions. This is built now because the voice agent (Component 3.2) needs it immediately. Records audio from any source (voice agent WebSocket stream, Meet recording file, uploaded audio file). Sends to Google Cloud Speech-to-Text (dynamic batch, $0.003/min). Produces timestamped transcript with speaker diarization. Stores transcript in Cloud SQL linked to the source (conversation_id for voice agent, session_id for therapy). Handles: multiple audio formats, variable length recordings, speaker identification, progressive transcription for long conversations. Quality validation: minimum word count, language detection, completeness check.
Triggered by: System initialization (infrastructure component)
Outputs: Transcription service available for any audio source, transcripts stored in Cloud SQL
Tech: Cloud Speech-to-Text dynamic batch API, Cloud SQL transcript storage, audio format handling, Drive storage for recordings
PHASE 3
Client Onboarding
The potential client creates an account. From the moment they speak, everything is recorded and tied to their record.
Component 3.1: Client Account Creation
Minimal signup: email and password. This creates the client record in Cloud SQL immediately with a unique client_id. Every piece of data collected from this point forward is tied to this ID. No information is required beyond email/password at this stage. The portal or marketing materials direct them here. After account creation, they land in the authenticated client portal where the voice agent is immediately available. The account creation event is logged and the client record enters state: "account_created".
Triggered by: Visitor clicks "Create Account" on landing page
Outputs: Client record in Cloud SQL (client_id, email, created_at, state=account_created), Firebase Auth account, authenticated portal access
Tech: Firebase Auth, Cloud SQL clients table, state machine initialization
Component 3.2: Voice Agent (Auth-Required, Recorded from Word One)
The voice agent is the first thing a new client interacts with after logging in. It requires authentication and every conversation is recorded and transcribed in real-time using the shared pipeline (Component 2.2). The agent operates in a fluid conversational mode that handles both inquiry and intake naturally. It can answer questions about the practice (from the practice profile), and it progressively collects client information as the conversation flows. There is no hard mode switch between "inquiry" and "intake." If someone asks about insurance, the agent answers and then naturally asks "Would you like me to check if your insurance covers sessions here?" The client can hang up and come back days later; the agent reloads their record and picks up where they left off. All conversation transcripts are stored as clinical artifacts tied to the client record. Data collected includes: full name (minimum for proceeding), date of birth, address, phone, emergency contact, presenting concerns, mental health history, medical history, current medications, prior therapy experience, substance use history, family/relationship context, goals for therapy. Each piece of data saved updates the client record and the intake completion percentage shown on the clinician dashboard.
Triggered by: Client logs in after account creation
Outputs: Recorded and transcribed conversations tied to client_id, progressively populated client record, intake completion percentage, conversation transcripts as clinical artifacts
Tech: Gemini Live API with client context loaded, shared recording pipeline (Component 2.2), Cloud SQL progressive saves, WebSocket audio streaming
Component 3.3: Insurance Card & ID Upload + AI Extraction
Client uploads photos of their insurance card (front/back) and photo ID through the portal or references them during voice agent conversation (agent provides upload link). Gemini Vision API extracts: from ID (full legal name, DOB, address), from insurance card (payer name, member ID, group number, plan type, copay/coinsurance info, phone numbers). Extracted data is presented to the client for confirmation/correction, then saved to the client record. If the client indicates cash pay during their voice agent conversation, they skip this entirely and the agent records their cash pay preference. The agent can prompt this step naturally: "To check your insurance coverage, I'll need a photo of your insurance card. I'm sending you a link to upload it now."
Triggered by: Client mentions insurance during voice agent conversation, or selects insurance option in portal
Outputs: Verified insurance demographics in client record, images stored in Drive, extraction confidence scores, OR cash pay status recorded
Tech: Gemini Vision API, Cloud Run image upload endpoint, Drive storage, Cloud SQL insurance_info table
Component 3.4: Eligibility Verification (VOB)
Automated verification of benefits. Triggered as soon as insurance card data is confirmed. For MVP/manual mode: creates a task in the billing task queue with payer phone number, member ID, and verification checklist (mental health benefits, IOP/PHP coverage, copay, coinsurance, deductible status, prior auth requirements, session limits, telehealth coverage, out-of-network benefits). For automated mode: Stedi eligibility API (270/271 transaction) returns structured benefit data. Results stored on client record. The voice agent can communicate basic results: "Great news, your Blue Cross plan does cover intensive outpatient programs." VOB completion is a critical trigger: it means we have a qualified lead with verified insurance, ready for the BD team.
Triggered by: Insurance card data confirmed by client (Component 3.3)
Outputs: VOB results on client record, qualified lead ready for BD handoff
Tech: Stedi 270/271 API (automated) or manual task queue (MVP), Cloud SQL benefits table
Component 3.5: Business Development Alert & Lead Package
When VOB is complete and sufficient intake info has been collected by the voice agent, the system assembles a warm lead package and alerts the BD team. The lead package includes: client name, contact info, presenting concerns and clinical history (from voice agent conversations), insurance verification results (payer, plan, IOP/PHP coverage details, copay/coinsurance, deductible status, prior auth requirements), voice agent conversation transcripts, intake completion status, and any red flags or special considerations the agent noted. BD receives notification via their preferred channel (email, SMS, dashboard alert, Slack). The lead appears on the BD pipeline dashboard with a status of "ready for outreach." This is the handoff from AI to human. The voice agent has qualified the lead and gathered the data; the BD person now brings the human judgment, rapport-building, and nuanced clinical assessment needed for IOP/PHP level-of-care determination.
Triggered by: VOB complete (Component 3.4) AND minimum intake data collected by voice agent
Outputs: Lead package assembled, BD team alerted, lead appears on BD pipeline dashboard with status "ready for outreach"
Tech: Cloud SQL lead pipeline tables, notification service (email/SMS/Slack webhook), BD dashboard UI, lead scoring logic
Component 3.6: BD Outreach & Level-of-Care Assessment (Human in the Loop)
The BD person calls the potential client using the lead package as context. They already know the client's name, what brought them to seek care, and their insurance situation. The BD person's role is to: build personal rapport and trust, assess appropriateness for IOP/PHP level of care (frequency, intensity, safety considerations), answer nuanced questions about the program (schedule, structure, what a typical day looks like, family involvement, medication management), assess the client's ability to commit to the program schedule, discuss financial responsibility (copays, deductible, estimated out-of-pocket), and address any barriers to admission. The BD person documents their assessment and notes in the system. If the client is a good fit and wants to proceed, the BD person fills out the Financial Agreement Request Form in the system. If the client is not appropriate for IOP/PHP, the BD person can document this and facilitate a referral to a more appropriate level of care. BD pipeline dashboard tracks: leads contacted, conversion rate, time to contact, reasons for non-conversion.
Triggered by: BD receives lead alert (Component 3.5)
Outputs: BD assessment notes on client record, client qualified or referred out, Financial Agreement Request Form submitted if proceeding
Tech: Cloud Run BD interface, Cloud SQL lead status tracking, financial agreement request form, referral documentation
Component 3.7: Financial Agreement Request Form Submission (Trigger)
The BD person submits the Financial Agreement Request Form, which captures: confirmed level of care (IOP or PHP), estimated length of stay, session frequency and schedule, confirmed insurance benefits and patient financial responsibility (copay per session, coinsurance percentage, deductible remaining, estimated total out-of-pocket), payment arrangement details (payment plan if needed), and any prior authorization requirements or status. Submitting this form is the critical trigger that initiates the formal admission process. It tells the system: this person has been qualified by both AI and a human, insurance is verified, financial terms are understood, and we are ready to send paperwork.
Triggered by: BD person completes assessment and client wants to proceed (Component 3.6)
Outputs: Financial agreement request record in Cloud SQL, TRIGGERS document generation and e-sign workflow
Tech: Cloud Run form UI, Cloud SQL admission pipeline, trigger to Component 3.8
Component 3.8: Document Generation & E-Signature
Triggered by the Financial Agreement Request Form submission (Component 3.7). System generates the full consent and admission document package from templates, pre-populated with all client data and the financial terms from the BD form: Informed Consent for Treatment, Notice of Privacy Practices (HIPAA), Financial Agreement (with specific copay/coinsurance/payment plan terms from BD form), Telehealth Consent (if applicable for virtual IOP), Program Agreement (IOP/PHP schedule commitment, attendance expectations, rules), Release of Information forms (if needed for care coordination), practice-specific forms. Documents sent to client for e-signature via the portal. Client receives notification with a clear call to action: "Your admission documents are ready to sign." First signature captures drawn/typed signature image, stored for future one-click signing. Each signed document: signature + metadata (timestamp, IP, user agent, document hash) captured, SHA-256 hash written to blockchain, Google Doc locked, signed PDF to Drive. All documents signed = client state advances to "admission_documents_signed" which triggers intake scheduling.
Triggered by: Financial Agreement Request Form submitted by BD (Component 3.7)
Outputs: Signed admission documents in Drive, blockchain transaction IDs, client state = admission_documents_signed, TRIGGERS intake scheduling
Tech: Google Docs API (template population), Cloud Run signing endpoint, blockchain write, Drive API (lock + PDF export)
Component 3.9: Intake Session Scheduling
Triggered when all admission documents are signed (Component 3.8). System creates an intake session appointment with a clinician. For MVP: BD person or intake coordinator selects clinician and time slot from available Calendar slots. System creates Calendar event, links it to the client record, and notifies both client and clinician. Pre-intake checklist confirms: all documents signed, insurance verified, prior auth obtained (if required), financial agreement executed. Client receives appointment confirmation with details: date, time, clinician name, how to join (Meet link for virtual IOP), what to expect in the intake session. Clinician receives intake package: all voice agent transcripts, BD notes, presenting concerns, insurance info, signed documents. The clinician walks into the intake session already knowing this person's story.
Triggered by: All admission documents signed by client (Component 3.8)
Outputs: Calendar event created, client and clinician notified, intake package delivered to clinician
Tech: Google Calendar API, Cloud Run notification service, Cloud SQL appointment linking, intake package assembly
PHASE 4
Clinical Session
The client has their appointment. The system captures the session using the same recording pipeline built in Phase 2.
Component 4.1: Session Recording Detection & Capture
For telehealth: clinician conducts session via Google Meet. Recording automatically saved to Drive when meeting ends. System detects new recording via Drive webhook or polling, matches it to a calendar event (by event ID or time window) and therefore to a client record. For in-person: clinician uses phone as microphone, uploads audio file to designated Drive folder, system matches by timestamp/filename convention. Multiple recording segments (from reconnects) are stitched together. Recording file stored in client's Drive folder. Uses the same recording infrastructure as the voice agent (Component 2.2) but triggered by Meet/upload rather than WebSocket stream.
Triggered by: Google Meet recording saved to Drive, or audio file uploaded to designated folder
Outputs: Recording file linked to session record in Cloud SQL, ready for transcription
Tech: Drive API webhooks/polling, Calendar API event matching, Cloud SQL session table, shared recording pipeline
Component 4.2: Session Transcription
Recording sent through the shared transcription pipeline (Component 2.2). Same Speech-to-Text service, same storage, same quality validation. The difference from voice agent transcription: therapy session transcripts are longer (45-60 min vs variable intake conversations), speaker diarization maps to clinician and client (vs agent and client), and the transcript feeds into clinical note generation rather than record population.
Triggered by: Recording file detected and linked to session (Component 4.1)
Outputs: Timestamped, speaker-diarized transcript in Cloud SQL, linked to session record
Tech: Shared transcription pipeline (Component 2.2), Cloud SQL session-transcript linking
Component 4.3: AI Clinical Note Generation
Gemini generates a clinical note from the session transcript. Note template varies by session type: Intake Assessment (90791) produces biopsychosocial assessment with history, mental status exam, diagnostic impressions, and treatment recommendations. The intake note is enriched by the voice agent intake transcripts (the AI has the full context of everything the client shared during onboarding). Progress Note (90834/90837) produces SOAP or DAP format with treatment plan progress, interventions used, client response, and updated goals. Group Note (IOP) produces per-client individual notes. The AI references the client's existing record (diagnoses, treatment plan, prior notes, intake agent conversations) for continuity. Note is created as Google Doc in client's Drive folder. Clinician notified that note is ready for review.
Triggered by: Session transcript complete and quality validated (Component 4.2)
Outputs: Draft clinical note as Google Doc, clinician notification, suggested CPT code based on session duration
Tech: Gemini API with clinical prompting, Google Docs API, Cloud SQL session metadata, note template system
Component 4.4: Clinician Note Review & Editing
Clinician opens AI-generated note in Google Docs and reviews it. They can edit freely: correct clinical details, adjust language, add observations the recording may have missed, update diagnoses or treatment plan elements. The note lives in Docs with familiar editing tools. Clinical data modified in the note (new diagnosis, treatment plan update) can be flagged for syncing back to the structured record. Clinician confirms session type (determines CPT code) and duration. System suggests CPT code: first session defaults to 90791, under 38 min = 90832, 38-52 min = 90834, 53+ min = 90837.
Triggered by: Clinician receives note-ready notification (Component 4.3)
Outputs: Reviewed and edited note in Google Docs, confirmed CPT code, ready for signing
Tech: Google Docs (native editing), Cloud Run dashboard with note queue, CPT suggestion logic
Component 4.5: Note Signing & Locking
Clinician clicks sign link (in dashboard or embedded in Doc). Cloud Run signing page displays read-only render of final Doc content. Clinician's stored signature image pre-loaded (captured on first use). One-click confirm: signature composited onto PDF, metadata captured (timestamp, IP, user agent), document hash generated (SHA-256), hash written to blockchain, Google Doc locked (edit permissions revoked via Drive API), signed PDF exported to Drive. If the clinician is a supervised associate: note enters "pending supervisor review" state instead of fully locking. Supervisor receives notification. The signed note is the trigger for the entire billing cascade.
Triggered by: Clinician finishes reviewing/editing note and clicks sign
Outputs: Signed locked note (PDF in Drive), blockchain transaction ID, audit log entry, BILLING TRIGGER
Tech: Cloud Run signing endpoint, blockchain write, Drive API (permissions + PDF export), Cloud SQL note_status update
Component 4.6: Supervisor Co-Signature (Conditional)
For supervised clinicians only. Supervisor receives notification of pending note. Opens note in Google Docs (read-only). Reviews clinical content, diagnostic accuracy, treatment appropriateness. Can request revision (unlocks Doc, returns to clinician with comments) or approve and co-sign. Co-signature: one-click confirm, signature added to PDF, new hash generated with both signatures, blockchain write. Note fully locked with two signatures. This is the billing trigger for supervised clinicians.
Triggered by: Clinician signs note while in supervised status (Component 4.5)
Outputs: Co-signed note with two signatures, fully locked, supervisor audit trail, BILLING TRIGGER
Tech: Same signing infrastructure as 4.5, supervisor role in RBAC, notification system
PHASE 5
Billing & Revenue Cycle
The signed note triggers the entire billing cascade. Sign your note, get paid.
Component 5.1: Claim Assembly
A signed (and co-signed if required) note automatically triggers claim assembly. System pulls: client demographics and insurance (from onboarding), provider NPI and taxonomy (from practice profile), CPT code (confirmed during note review), ICD-10 diagnosis codes (from treatment plan/note), date of service (from calendar event), place of service (telehealth vs office), fee amount (from fee schedule by CPT + payer), referring provider and prior auth number (if applicable). Pre-submission scrubbing validates all fields, checks CPT/ICD compatibility, verifies payer-specific requirements, flags issues (e.g., billing 90791 for a client who already had one this year). Output: structured claim record ready for submission.
Triggered by: Note signed and locked (Component 4.5 or 4.6)
Outputs: Structured claim record in Cloud SQL, scrubbing results, ready for submission
Tech: Cloud SQL claim assembly logic, fee schedule lookup, payer rules engine, CPT/ICD validation
Component 5.2: CMS-1500 Generation & Payer Instructions (Manual Fallback)
For practices without automated clearinghouse: generates completed CMS-1500 form as PDF with all claim data pre-populated, plus payer-specific submission instructions (portal link, copy-paste-ready data). Stored in Drive. This is the MVP billing workflow that works from day one without Stedi. Even manually, the system has done 90% of the work. The human just submits through the portal.
Triggered by: Claim assembled and scrubbed (Component 5.1), practice not on automated billing plan
Outputs: CMS-1500 PDF in Drive, payer portal instructions, task in billing queue
Tech: PDF generation library, CMS-1500 template, payer portal database, Drive storage
Component 5.3: Automated Claim Submission (Clearinghouse) [$2/claim]
For practices on the billing plan: claim automatically submitted through Stedi. System generates ANSI X12 837P transaction, submits via Stedi API, receives acknowledgment. Full RCM included: submission, status tracking, denial management, payment posting, patient billing. This is the "sign your note, get paid" promise. Clinician does nothing after signing. Stedi handles payer connectivity and HIPAA-compliant transmission. System monitors for rejections and auto-corrects common issues before resubmitting. $2 per claim covers the entire revenue cycle for that claim from submission to payment posted.
Triggered by: Claim assembled and scrubbed (Component 5.1), practice on billing plan
Outputs: 837P submitted to payer via Stedi, claim status tracking initiated, $2 charge recorded
Tech: Stedi 837P API, Cloud SQL claim_status tracking, auto-correction rules, usage metering
Component 5.4: Claim Status Tracking & Denial Management
Continuous monitoring of submitted claims. Automated: Stedi 277 responses parsed and stored. Manual: task queue prompts follow-up at configurable intervals. Claim states: submitted, accepted, pending, paid, denied, rejected, appealed. Denials categorized by reason code with suggested corrective action: missing modifier (auto-add and resubmit), timely filing (flag urgent), medical necessity (AI-generated appeal letter using clinical note content), coordination of benefits (identify primary/secondary issue). Dashboard shows claim aging, denial rates by payer, revenue pipeline.
Triggered by: Claim submitted (Component 5.2 or 5.3), status update received or follow-up interval reached
Outputs: Updated claim status, denial categorization, suggested actions, appeal drafts
Tech: Stedi 277 API, Cloud SQL claim lifecycle tracking, Gemini for appeal generation, dashboard analytics
Component 5.5: Payment Posting (ERA/EOB Processing)
When payment arrives: Automated: Stedi 835 parsed. Payment amount, adjustments, patient responsibility, and denial reasons extracted and posted to client ledger. Manual: staff enters payment from EOB. System calculates: allowed amount vs billed, insurance payment received, contractual adjustment, patient responsibility (copay, coinsurance, deductible). If payment differs from contracted rate, flags for review or auto-appeal.
Triggered by: Payment received from payer (835 transaction or manual entry)
Outputs: Updated client ledger, patient responsibility calculated, underpayment flags
Tech: Stedi 835 API, Cloud SQL ledger table, contracted rate comparison, auto-reconciliation
Component 5.6: Client Ledger & Patient Billing
Running financial record per client: charges, insurance payments, adjustments, patient payments, balance. When patient responsibility is determined, system generates statement. Client views balance and pays through portal via Stripe (credit/debit, FSA/HSA, payment plans). Automatic statement generation, past-due reminders. Superbill generation for out-of-network clients.
Triggered by: ERA/EOB processed (Component 5.5) or cash-pay session completed
Outputs: Client ledger in Cloud SQL, patient statements, Stripe payment collection, superbills
Tech: Cloud SQL ledger tables, Stripe integration, PDF statement generation, client portal payment UI

Component 6.1: Client Portal (Authenticated)
The logged-in client experience. Dashboard shows: voice agent access (always available), upcoming appointments (from Calendar), documents pending signature, signed documents archive, billing balance and payment history, intake completion status (for clients still onboarding), conversation history with voice agent. Clients can: talk to the voice agent, update insurance info, upload new card images, make payments, view and sign documents, book appointments. Mobile-responsive PWA.

PHASE 7
IOP / Group Practice Extensions
Components for the virtual IOP pilot and group practice scenarios.
Component 7.1: Group Session Management
IOP groups: multiple clients in one Meet session. System tracks attendance per client per group, records once but generates individual notes per client, handles late joins/early departures, tracks group size for billing compliance. Calendar event linked to multiple client records. Attendance confirmed by clinician or detected from Meet participant logs. Each client gets individual session record linked to group event.
Triggered by: Group therapy session scheduled on Calendar with multiple clients
Outputs: Per-client attendance records, group session metadata, individual session records per attendee
Tech: Cloud SQL group_session and session_attendance tables, Calendar API, Meet participant API
Component 7.2: Per-Client Group Note Generation
One recording, one transcript, individual notes per client. Gemini generates individual notes by: identifying each client's contributions via speaker diarization, cross-referencing each client's treatment plan, documenting interventions and individual responses, noting clinical observations specific to each person. Each note is a separate Google Doc in that client's folder. Most complex AI generation task in the system.
Triggered by: Group session transcript complete with speaker diarization
Outputs: Individual draft notes per client as Google Docs, each referencing that client's treatment plan
Tech: Gemini with per-client context loading, speaker-to-client mapping, Google Docs API batch creation
Component 7.3: Multi-Clinician Scheduling & Census
IOP programs with multiple clinicians. Unified schedule view across all clinicians' calendars, group assignments, client assignments. Census tracking: active clients in program, attendance rates, discharge planning timeline. Shift coordination for co-facilitated groups.
Triggered by: IOP program setup with multiple clinicians
Outputs: Unified schedule view, census tracking, clinician assignment management
Tech: Google Calendar API (multiple calendars), Cloud SQL program management, dashboard views
Component 7.4: Prior Authorization & Continued Stay Reviews
IOP often requires prior auth and periodic continued stay reviews. System tracks: initial auth dates and approved sessions, auth expiration, review due dates. When review due, Gemini drafts clinical summary supporting medical necessity from recent notes, attendance, and treatment progress. Clinician reviews and submits. Alerts when approaching session limits.
Triggered by: Auth tracking initiated at IOP admission, review intervals per payer
Outputs: Auth status tracking, review due date alerts, AI-drafted clinical summaries
Tech: Cloud SQL authorization tables, Gemini for clinical summaries, payer-specific review rules
Component 7.5: Discharge Planning & Aftercare
Client completes IOP or steps down. Discharge summary generated from full treatment history: admission, presenting concerns, diagnoses, treatment provided, progress, disposition, aftercare recommendations. Gemini drafts from notes, treatment plan, and attendance data. Referral coordination. Final billing reconciliation ensures all sessions billed and payments posted before case closure.
Triggered by: Clinical decision to discharge or payer auth ending
Outputs: Discharge summary (signed PDF), aftercare plan, final billing reconciliation, case closure
Tech: Gemini for summary generation, all prior components for data aggregation, Cloud SQL case management

Premium Services & Upsells
Revenue-generating services layered on the core platform.

Assist clinicians in getting credentialed with insurance payers. Practice profile data pre-populates CAQH ProView fields. Track 120-day attestation deadlines. Payer application tracking per payer with status, follow-ups, missing docs. Contract/fee schedule review against benchmarks. Re-credentialing (2-3 year cycles) monitored automatically. Triggered by: Clinician signs up for credentialing service (premium upsell)

Component 8.2: Security & Compliance Auditing
Automated security assessment of the practice's GCP project: IAM roles, API access patterns, unusual PHI access, encryption status, firewall rules. Generates HIPAA compliance reports for auditors, insurance panels, or liability carriers. Periodic scans with anomaly alerts. Annual compliance summary.
Triggered by: Practice subscribes to audit service or periodic scan triggers
Outputs: Security reports, HIPAA compliance documentation, anomaly alerts
Tech: GCP Security Command Center API, audit log analysis, automated report generation
Component 8.3: Custom Voice Agent Tuning
Specialty-specific customization of the voice agent. Trauma-informed intake (pacing, screening), substance use assessment (CAGE, AUDIT), child/adolescent (parent/guardian workflow), couples (individual + joint). Conversational screening tools (PHQ-9, GAD-7, PCL-5). Practice personality and tone matching. Consulting engagement with the clinician to design their ideal flow.
Triggered by: Clinician requests customization (premium upsell)

Tech: Gemini Live API prompt engineering, Cloud SQL custom flow config, screening tool integration
Component 8.4: Analytics & Business Intelligence
Practice performance: inquiry-to-intake conversion, no-show rates, sessions per client, revenue per client, revenue per CPT, payer mix, denial rate by payer, days to payment, revenue trends. For IOP: census trends, length of stay, readmission rates, attendance patterns, auth utilization.
Triggered by: Sufficient data accumulated (1-3 months of operation)
Outputs: Interactive dashboard, periodic reports, benchmark comparisons
Tech: Cloud SQL aggregation, dashboard UI, scheduled reports, benchmark database
Component 8.5: Specialized Agent Teams
Dedicated AI agents beyond intake: Scheduling Agent (reschedules, cancellations, waitlist, no-show follow-up), Billing Agent (client balance questions, EOB explanations, payment arrangements), Referral Agent (incoming referrals, coordination). Each agent has role-specific data access enforced by RBAC.
Triggered by: Practice identifies need for additional automation (premium upsell)
Outputs: Specialized agents handling practice functions autonomously
Tech: Gemini Live API per agent, role-specific data access, task management, conversation logging
